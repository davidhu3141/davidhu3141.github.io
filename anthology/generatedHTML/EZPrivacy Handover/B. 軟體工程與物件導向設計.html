軟體開發過程 Software Development Process

四個基本開發活動

Software Specification

  • 開出規格，定義這項專案的目的與最終目標、要交付怎樣的成品

Software Design and Implementation

  • Design: 規劃專案藍圖、軟體架構、人力分配等，有許多工具可以輔助（Computer Aided Software Engineering）
  • Implementation: 實作、寫文件。也就是實體的產出。

Software Validation

  • 檢驗產品，確認產品能夠滿足使用者需求

Software Evolution

  • 產品需要更新以符合需求的變遷。變遷總是無法避免的

開發過程模型 Process Models

開發模型幾乎都會滿足以上四個基本步驟，但是時程的規劃、開發的哲學大有不同。這些明確的進程設計就是 Process Models of Software Development

成功的軟體工程師能夠為專案選擇適當的開發模型

瀑布模型 Waterfall Model

  • 在第一個步驟 Requirement 完全規劃好之後開發的所有事，如交付成品的規格、軟體開發的架構、人力、時程的安排等。執行時也是一個步驟完全完成之後才進行下一個。

  • 如果有任何一個步驟執行不下去或是發現規劃錯誤，都必須從頭規劃

特色與用途

  • 非常缺乏彈性，需求一改就必須重新來過

  • 適合開發需求變動很小的大型程式。軟體開發的風險、成本、成品交付的時間可以輕易被評估。需求變遷仍是無法避免的

例子

  • 金融業軟體：跑在老舊機器上、難以重新學習或重新安裝、資安需求高、規模龐大、需求變動小。
  • IBM 等全球化且規模大的公司

修正的瀑布模型

  • 某個階段的失敗，允許回到上一個階段，而非從頭來過

迭代增量模型 Iterative & Incremental Model

迭代與增量模型通常透過重複執行瀑布模型，藉由重複的「規劃、開發、交付、規劃…」來達到目的。通常會將兩者混用。

這兩個模型也可以視為瀑布模型的相反。

Refer to: http://kojenchieh.pixnet.net/blog/post/395295317

增量開發

  • 每次交付成品新的一部份、新功能。
  • 每次交付後可以重新調整開發進度與方向

迭代開發

  • 每個交付的版本都可以上線
  • 方便掌握客戶需求
  • 對小開發團隊而言比較實際，交付一個版本才有後續資金湧入，並且以開發心得與使用者回饋來調整開發方向。

迭代與增量模式為敏捷開發的核心概念之一

除了以上四個模型之外還有許多其他的模型，例如 Spiral Model，Component Based，Reuse-Oriented

敏捷軟體開發方法 Agile software development

有些概念不是明確的進程設計，而是提供方法論與開發準則。有些準則適用於大型的開發，如 RUP；有些方法則適合輕量級的專案，如敏捷軟體開發

有時客戶的需求改變很快，不太可能產出完全穩定版，這就是以往計劃導向策略的困境。完整而長遠的設計規劃，難以因應現實的變遷。

方法與特徵

  1. 用 Incremental 與 Iterative 來規劃與交付，每次交付的增量很小
  2. 裁撤需要長時間才能完成的工作，一切簡化，但簡化有時候需要額外的工作
  3. 專注於軟體開發本身、以非正式的溝通取代不必要的文件
  4. 客戶也參與開發過程，直接在開發過程中提出要求，取得頻繁的使用者回饋
  5. 價值與風險排序

開發者或是開發團隊間要有良好的溝通
適用中小型專案、生命周期短的系統

不適合的專案

資安需求高的專案、不熱心提供意見的客戶、企業文化難以改變的公司、強烈需要預測專案動向的公司

物件導向設計

使用物件導向哲學來開發程式，有許多準則可以幫助我們。

SOLID 準則

SOLID 準則是五個常用的準則的縮寫

單一功能原則

一個類別應該只提供一項封裝好的功能

開閉原則

一個撰寫完成的類別總是開放大家擴充，但是不開放別人修改（開放外部擴充，封閉修改內部）

里氏替換原則

若 是 的子類別，則 應該滿足

對於所有接受 型態的參數的函數 ，也仍然應該正常運行

介面隔離原則

儘量將介面切細、專一化，避免不需要的實作或依賴

依賴反轉原則

一種降低程式碼耦合性的手段，用以達成下列兩項原則

  • 實作的程式依賴於抽象的介面，反之不可成立
  • 上層的模組不應該依賴與底層的模組，兩者應該依賴於抽象介面
